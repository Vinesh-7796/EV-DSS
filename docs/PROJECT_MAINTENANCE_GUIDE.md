# EV-DDSS Project Maintenance Guide

This guide documents the implementation that exists in this repository as inspected and tested on 2026-07-06. It covers the AI core in `ev-ddss/`, the application backend in `backend/`, and the React frontend in `frontend/`.

## Verified Project Shape

The runnable application has two layers:

- `ev-ddss/`: AI core, document processing, retrieval, reasoning, validation, configuration, and CLI.
- `backend/`: FastAPI application layer that imports `ev-ddss/` by inserting it into `sys.path`.
- `frontend/`: Vite React UI.
- `start.bat`: canonical Windows all-in-one launcher.
- `combined_starter.py`: duplicate launcher marked by its own header as "use start.bat instead on Windows"; treat it as deprecated.

Use the project virtual environment, not bare `python`:

```powershell
cd "D:\Github\EV diagnostics AI_POC\ev-ddss"
.\.venv\Scripts\python.exe main.py --help
```

On this machine, bare `python` is Python 3.14.3 and did not have `typer`; the project venv is Python 3.14.2 and successfully exposes the EV-DDSS CLI.

## Canonical Commands

From `D:\Github\EV diagnostics AI_POC\ev-ddss`:

```powershell
.\.venv\Scripts\python.exe main.py pipeline --path data/raw
.\.venv\Scripts\python.exe main.py retrieval-build --path data/store/json
$env:PYTHONIOENCODING="utf-8"; .\.venv\Scripts\python.exe main.py retrieval-query P0A00 --top-k 3 --no-log
```

The `pipeline` command scans raw documents, parses them, writes `data/processed/<type>/<stem>.json`, writes `data/store/json/<type>/<stem>.json`, and updates `data/store/json/manifest.json`.

The `retrieval-build` command loads documents from `data/store/json`, warms the embedding model, attempts to connect to Qdrant, upserts vector chunks into collection `semantic_chunks`, loads the graph from `data/knowledge/canonical_entity_index.json`, loads image metadata from `data/store/images`, and checks PostgreSQL availability.

## Supported Document Types

Supported extensions are registered in `ev-ddss/processing/engine.py` and `ev-ddss/ingestion/pipeline/pipeline.py`:

| Type | Extensions | Processor |
|---|---|---|
| PDF | `.pdf` | `processing/pdf/processor.py` |
| Excel | `.xlsx`, `.xls` | `processing/excel/processor.py` |
| DBC | `.dbc` | `processing/dbc/processor.py` |
| Images | `.png`, `.jpg`, `.jpeg`, `.tiff`, `.tif`, `.bmp` | `processing/image/processor.py` |

Put new or changed files in:

```text
ev-ddss/data/raw/
```

## Document Update Workflow

### PDF

| Question | Actual Implementation |
|---|---|
| Folder | `ev-ddss/data/raw/*.pdf` |
| Scripts | `main.py pipeline`, then `main.py retrieval-build` |
| Order | pipeline first, retrieval-build second |
| JSON rebuilt | Yes: `data/processed/pdf/<stem>.json`, `data/store/json/pdf/<stem>.json`, `data/store/json/manifest.json` |
| Vector rebuilt | Attempted by `retrieval-build` through Qdrant collection `semantic_chunks` |
| Graph rebuilt | No CLI rebuild exists. `retrieval-build` only loads `data/knowledge/canonical_entity_index.json` |
| Files generated | Processed JSON and store JSON |
| Backend restart | Recommended after graph/config/model changes. JSON fallback reloads files per query, but backend holds initialized engines |
| Frontend restart | No |
| Caches | No explicit app cache. Restart backend to clear `get_settings()` singleton and in-memory engines |
| Incremental/full | JSON is overwritten per stem. Qdrant upserts are incremental but use non-global chunk IDs, so clean full vector rebuild is safer |

Flow:

```text
PDF changed in data/raw
  ↓
.\.venv\Scripts\python.exe main.py pipeline --path data/raw
  ↓
data/processed/pdf and data/store/json/pdf updated
  ↓
clear Qdrant collection if doing a clean vector rebuild
  ↓
.\.venv\Scripts\python.exe main.py retrieval-build --path data/store/json
  ↓
restart backend if it was running
  ↓
ready
```

## Clean Vector Rebuild

There is no CLI command to drop the Qdrant collection. If Qdrant is running, use this before `retrieval-build` for a clean corpus rebuild:

```powershell
cd "D:\Github\EV diagnostics AI_POC\ev-ddss"
.\.venv\Scripts\python.exe -c "from retrieval.vector_index import VectorIndex; v=VectorIndex(); v.connect(); print(v.delete_collection())"
.\.venv\Scripts\python.exe main.py retrieval-build --path data/store/json
```

This is recommended because `retrieval/chunking.py` resets chunk IDs per document (`CHUNK00001`, etc.), so repeated upserts can collide.

## Model Replacement Workflow

### Files involved

| Purpose | File |
|---|---|
| Runtime/model config defaults | `ev-ddss/config/config.yaml` |
| Pydantic settings | `ev-ddss/config/config.py` |
| Reasoning config resolution | `ev-ddss/reasoning/config.py` |
| Adapter selection | `ev-ddss/reasoning/engine.py` |
| Ollama HTTP calls | `ev-ddss/reasoning/adapters/ollama_adapter.py` |
| Backend imports and settings load | `backend/app.py`, `backend/services/ai_service.py` |
| Read-only config endpoint | `backend/routes/configuration.py` |

Runtime is selected by:

```yaml
reasoning:
  runtime: ${REASONING_RUNTIME:-ollama}
```

Model is selected by:

```yaml
reasoning:
  model: ${REASONING_MODEL:-qwen3:8b}
```

Adapter selection is in `ReasoningEngine._create_adapter()`:

| `REASONING_RUNTIME` | Adapter |
|---|---|
| `ollama` | `OllamaAdapter` |
| `openai` | `OpenAICompatibleAdapter` |
| `vllm` | `VLLMAdapter` |

Ollama is called in `OllamaAdapter`:

- Health/list models: `GET {OLLAMA_URL}/api/tags`
- Generation: `POST {OLLAMA_URL}/api/chat`

### Replace Qwen3 with another Ollama model

No application logic needs changing. Set `REASONING_MODEL` to an installed Ollama tag and restart the backend.

Examples:

```powershell
$env:REASONING_RUNTIME="ollama"
$env:REASONING_MODEL="llama3.1:8b"
```

```powershell
$env:REASONING_MODEL="mistral:7b"
$env:REASONING_MODEL="gemma3:4b"
$env:REASONING_MODEL="deepseek-r1:8b"
$env:REASONING_MODEL="phi4:latest"
```

Then restart the backend. Frontend restart is not required unless the UI code changed. Reindexing is not required for LLM-only model replacement because embeddings and retrieval indexes are separate.

Verified simulation:

```powershell
$env:REASONING_MODEL="llama3.1:8b"
$env:PYTHONPATH="D:\Github\EV diagnostics AI_POC\ev-ddss"
.\.venv\Scripts\python.exe -c "from reasoning.config import ReasoningConfig; cfg=ReasoningConfig().resolve(); print(cfg.runtime); print(cfg.model); from reasoning.engine import ReasoningEngine; e=ReasoningEngine(config=cfg); print(type(e._create_adapter()).__name__)"
```

Observed output:

```text

llama3.1:8b
OllamaAdapter
```

`ev-ddss/.env.example` does not list `REASONING_RUNTIME`, `REASONING_MODEL`, or `OLLAMA_URL`, even though the code uses them. Add those variables to your local `.env` manually when changing models.

## Embedding Model

Current retrieval embedding model:

| Property | Value |
|---|---|
| Model | `all-MiniLM-L6-v2` |
| Runtime | `sentence-transformers` loaded locally |
| Configuration file | `ev-ddss/config/config.yaml` under `retrieval.embedding_model` |
| Settings class | `RetrievalSettings.embedding_model` in `ev-ddss/config/config.py` |
| Loader | `EmbeddingGenerator` in `ev-ddss/retrieval/embeddings.py` |
| Vector dimension | `384` by default, then updated from the loaded model |
| Device | `settings.embedding.device`, default `cpu` |

To change it:

```powershell
$env:EMBEDDING_MODEL="sentence-transformers/all-mpnet-base-v2"
$env:EMBEDDING_DIMENSION="768"
```

Then rebuild vectors:

```powershell
cd "D:\Github\EV diagnostics AI_POC\ev-ddss"
.\.venv\Scripts\python.exe -c "from retrieval.vector_index import VectorIndex; v=VectorIndex(); v.connect(); print(v.delete_collection())"
.\.venv\Scripts\python.exe main.py retrieval-build --path data/store/json
```

Embedding replacement effects:

| Artifact | Must regenerate? | Why |
|---|---:|---|
| Vectors/Qdrant | Yes | Vector dimensions and embedding space change |
| JSON store | No, unless documents changed | JSON contains parsed content, not vectors |
| Graph | No, unless entity extraction logic/content changed | Graph does not depend on embedding vectors |
| Frontend | No | UI does not load embeddings |
| Backend restart | Yes | Settings and engines are initialized in backend process |
